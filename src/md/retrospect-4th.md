# 네 번째 회고

### 1. 텍스트가 아닌 '흐름'을 배우다

이번 주는 강사님과 함께 실제 웹사이트의 UI를 구현해보는 **실전 실습(Live Coding)** 위주로 진행되었다. 단순히 태그와 속성을 배우는 단계를 넘어, **"전문가는 어떤 순서로 사고하고 코드를 작성하는가"**를 엿볼 수 있는 시간이었다.

|     구분      | 이전의 나 (Past)                               | 현재의 나 (Now)                                          |
| :-----------: | :--------------------------------------------- | :------------------------------------------------------- |
| **접근 방식** | 눈에 보이는 스타일(색상, 폰트)부터 적용하려 함 | **큰 덩어리(Layout)**를 먼저 잡고 세부 디테일로 들어감   |
| **작업 순서** | `CSS` 작성하다가 `HTML` 수정하기 반복          | **`구조 설계` → `배치` → `스타일링`**의 효율적 흐름 체득 |

### 2. 클래스 네이밍과 재사용성

실습을 진행하며 단순히 '동작하는 코드'가 아니라 **'관리하기 좋은 코드'**에 대한 고민을 시작했다.

- **Observation**: 강사님이 공통 스타일을 유틸리티 클래스로 정의하고 재사용하는 패턴 관찰.
- **Learning**: CSS 설계가 잘 되어야 유지보수가 쉬워진다는 사실을 체감.

## 🚧 마주한 벽과 극복 (Trouble Shooting)

### 🛑 문제: '타이피스트'의 함정

강사님의 라이브 코딩을 따라 치는 시간은 몰입도가 높았지만, 자칫하면 생각 없이 손만 움직이는 '타이피스트'가 될 위험을 감지했다. 수업 중간중간 *"왜 여기서 flex 대신 grid를 썼을까?"*를 스스로 질문하지 않고, 단순히 에러 없이 따라가는 데 급급했던 순간들이 있었다.

### ✅ 해결: '복기'의 시간 가지기

따라 친 코드는 내 실력이 아님을 인정하고, 다음과 같은 루틴을 적용하기로 했다.

1.  수업 중에는 흐름과 **'Why'**에 집중하기.
2.  막히는 부분이 진짜 내 약점임을 인정하고 그 부분 집중 공략.
